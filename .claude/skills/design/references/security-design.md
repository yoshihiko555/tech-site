# セキュリティ設計

このドキュメントは `/design` スキルの拡張設計トラックとして、ユーザーの要望または AI の提案時に参照される。

## 目的

実装前にセキュリティアーキテクチャを策定し、脅威への対策方針を明確にする。
Secure-by-Design の考え方に基づき、設計段階でセキュリティを組み込む。

## 前提

- 要件定義（`functional.md`, `non-functional.md`）が存在すること
- 基本設計（`architecture.md`）が存在する、または並行して進行中であること

## いつ実施するか

以下のいずれかに該当する場合に実施を検討する：

- ユーザーが「セキュリティ設計」「脅威分析」「認証設計」等を明示した場合
- 個人情報・決済情報・医療情報等の機密データを扱うプロジェクト
- マルチテナントシステムや権限モデルが複雑なプロジェクト
- 外部 API 連携が多いプロジェクト
- 非機能要件でセキュリティが重点項目になっている場合

## ステップ

### Step 1: セキュリティ要件の整理

非機能要件（`non-functional.md`）からセキュリティ関連の要件を抽出し、不足があれば追加でヒアリングする。

#### ヒアリング項目

- 扱うデータの機密レベル（個人情報、決済情報、医療情報等）
- コンプライアンス要件（個人情報保護法、GDPR、PCI DSS、HIPAA 等）
- 想定する脅威の種類（外部攻撃、内部不正、データ漏洩等）
- セキュリティ監査の要否

#### 出力

`docs/security/security-requirements.md` に記載する。

```markdown
# セキュリティ要件

## データ分類

| 分類 | 該当データ | 保護レベル |
|------|----------|----------|
| 機密 | {例: パスワード、決済情報} | 暗号化必須、アクセスログ必須 |
| 内部 | {例: ユーザーメールアドレス} | アクセス制御必須 |
| 公開 | {例: 商品情報} | 改ざん防止 |

## コンプライアンス要件

| 規制 | 対象 | 主な要件 |
|------|------|---------|
| {規制名} | {対象データ/範囲} | {要件概要} |

## セキュリティ目標

- {目標1: 例「認証情報の漏洩を防止する」}
- {目標2: 例「権限を超えたデータアクセスを防止する」}
- {目標3: 例「監査証跡を記録する」}
```

---

### Step 2: 脅威モデリング

システムに対する脅威を体系的に分析する。

#### 設計内容

STRIDE モデルを参考に、主要コンポーネントごとの脅威を分析する：

| カテゴリ | 説明 | 例 |
|---------|------|-----|
| **S**poofing（なりすまし） | 認証の偽装 | セッションハイジャック |
| **T**ampering（改ざん） | データの不正変更 | リクエスト改ざん |
| **R**epudiation（否認） | 行為の否認 | ログ不足 |
| **I**nformation Disclosure（情報漏洩） | 不正な情報取得 | API レスポンスの過剰露出 |
| **D**enial of Service（サービス拒否） | サービス妨害 | レートリミット未設定 |
| **E**levation of Privilege（権限昇格） | 不正な権限取得 | IDOR、権限チェック漏れ |

#### 対話の進め方

1. `docs/architecture/architecture.md` のコンポーネント構成を確認
2. 各コンポーネント間の通信を洗い出す（信頼境界の特定）
3. 「このコンポーネントに対して、どんな攻撃が考えられますか？」と脅威を列挙
4. 脅威ごとにリスクレベル（高/中/低）と対策方針を決める

#### 出力

`docs/security/threat-model.md` に記載する。

```markdown
# 脅威モデル

## 信頼境界

{テキストベースの構成図に信頼境界を追記}

## 脅威一覧

| ID | コンポーネント | 脅威（STRIDE） | 説明 | リスク | 対策 |
|----|-------------|---------------|------|--------|------|
| TH-001 | {コンポーネント} | Spoofing | {説明} | 高 | {対策} |
| TH-002 | {コンポーネント} | Information Disclosure | {説明} | 中 | {対策} |

## リスクマトリクス

| リスクレベル | 対応方針 |
|------------|---------|
| 高 | 設計段階で対策を確定、実装必須 |
| 中 | 対策方針を決定、実装時に詳細化 |
| 低 | 認識のみ、必要に応じて対策 |
```

---

### Step 3: 認証・認可アーキテクチャ

ユーザー認証と権限管理の設計を行う。

#### ヒアリング項目

- 認証方式の希望（JWT、セッション、OAuth 2.0、SAML 等）
- ソーシャルログインの要否（Google、GitHub 等）
- MFA（多要素認証）の要否
- ユーザーロールと権限の種類
- API キー認証の要否（外部連携用）

#### 設計内容

- 認証フロー（ログイン、トークンリフレッシュ、ログアウト）
- 認可モデル（RBAC、ABAC 等）
- ロールと権限のマッピング
- トークン管理（有効期限、リフレッシュ戦略）

#### 出力

`docs/security/auth-design.md` に記載する。

```markdown
# 認証・認可設計

## 認証方式

- **方式**: {JWT / セッション / OAuth 2.0 等}
- **トークン有効期限**: アクセストークン {期間}、リフレッシュトークン {期間}
- **トークン保存先**: {httpOnly Cookie / localStorage 等}
- **MFA**: {要否と方式}

## 認証フロー

### ログイン

{テキストベースのシーケンス}

### トークンリフレッシュ

{テキストベースのシーケンス}

### ログアウト

{テキストベースのシーケンス}

## 認可モデル

### ロール定義

| ロール | 説明 | 権限概要 |
|--------|------|---------|
| {ロール名} | {説明} | {権限概要} |

### 権限マトリクス

| リソース | 操作 | {ロール1} | {ロール2} | {ロール3} |
|---------|------|----------|----------|----------|
| {リソース} | 読み取り | o | o | o |
| {リソース} | 作成 | o | o | x |
| {リソース} | 更新 | o | x | x |
| {リソース} | 削除 | o | x | x |

## パスワードポリシー

| 項目 | ルール |
|------|--------|
| 最小文字数 | {文字数} |
| 複雑さ | {ルール} |
| ハッシュアルゴリズム | {bcrypt / argon2 等} |
| ストレッチング回数 | {回数} |
```

### Step 4: データ保護設計

機密データの保護方針を定義する。

#### 設計内容

- 暗号化方針（通信時、保存時）
- 個人情報の取り扱い（収集、保存、削除）
- ログに含めてはいけない情報の定義
- データ保持期間と削除ポリシー

#### 出力

`docs/security/data-protection.md` に記載する。

```markdown
# データ保護設計

## 暗号化

| 対象 | 方式 | 備考 |
|------|------|------|
| 通信 | TLS 1.3 | 全通信を HTTPS 化 |
| パスワード | {ハッシュ方式} | ソルト付き |
| 機密データ（保存時） | {暗号化方式} | {鍵管理方式} |

## 個人情報の取り扱い

| データ | 収集目的 | 保持期間 | 削除方針 |
|--------|---------|---------|---------|
| {データ} | {目的} | {期間} | {方針} |

## ログポリシー

### ログに含めてはいけない情報

- パスワード（平文・ハッシュ問わず）
- アクセストークン・リフレッシュトークン
- クレジットカード番号
- {その他の機密情報}

### 監査ログ

| イベント | 記録項目 | 保持期間 |
|---------|---------|---------|
| ログイン成功/失敗 | ユーザーID、IP、タイムスタンプ | {期間} |
| 権限変更 | 実行者、対象者、変更内容 | {期間} |
| データ削除 | 実行者、対象データ | {期間} |
```

## サブエージェント活用

セキュリティ設計では `auth-designer` エージェントを活用できる：

```
Task(subagent_type="auth-designer", prompt="""
以下のシステムの認証・認可アーキテクチャを評価してください：

システム概要: {概要}
認証方式: {方式}
ロール構成: {ロール}

以下を分析してください：
1. 認証フローのセキュリティ上の問題点
2. 認可モデルの適切さ
3. 推奨される改善点
""")
```

## 注意事項

- セキュリティ設計の深さはプロジェクトのリスクレベルに応じて調整する
- 一般的な Web サービスでは Step 1（要件整理）+ Step 3（認証・認可）で十分なケースが多い
- 金融・医療・個人情報を大量に扱う場合は Step 2（脅威モデリング）+ Step 4（データ保護）まで実施する
- セキュリティ設計は実装後の `security-reviewer` によるレビューと組み合わせて効果を発揮する
- OWASP Top 10 は常に意識するが、設計書に全項目を網羅する必要はない — プロジェクトに関連する脅威に集中する
