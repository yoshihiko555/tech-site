# 基本設計フェーズ（Phase 2）詳細手順

このドキュメントは `/design` スキルの Phase 2 で参照される。

## 前提

Phase 1 の要件定義が完了しているか、ユーザーが既に要件を把握していること。
`docs/` 配下のドキュメント（`project-overview.md`, `glossary.md`, `requirements/`）が存在する場合はまず読み込んで文脈を把握する。

## 目的

要件に基づいて、システムの全体像を設計する。
「何を作るか」が決まった状態から「どう作るか」の方針と一覧を決める。
このフェーズでは「一覧」と「方針」を作成する。個別の詳細仕様は Phase 3 で扱う。

## ステップ

プロジェクトの性質によって必要なステップは異なる。
全ステップが必須ではなく、ユーザーと相談しながら必要なものを選択する。

### Step 1: アーキテクチャ設計

システム全体の構成を設計する。ほぼ全てのプロジェクトで必要。

#### ヒアリング項目

- 技術スタックの希望・制約（言語、フレームワーク）
- デプロイ先の想定（クラウド、オンプレ、ローカル）
- 既存システムとの連携要否
- ネイティブアプリの場合：クロスプラットフォーム（Flutter/React Native）か ネイティブ（Swift/Kotlin）か

#### 設計内容

- システム構成図（テキストベース）
- 主要コンポーネントとその責務
- 技術スタックの選定と理由
- コンポーネント間の通信方式
- ネイティブアプリの場合：アプリ⇔サーバー間の通信方式、ローカルストレージ戦略

#### サブエージェント活用

技術スタックの選定やアーキテクチャの妥当性検証には `architect` エージェントを活用できる：

```
Task(subagent_type="architect", prompt="""
以下の要件に対するアーキテクチャを評価してください：

要件概要: {要件サマリ}
提案アーキテクチャ: {設計案}

以下を分析してください：
1. アーキテクチャの妥当性
2. 代替案との比較
3. リスクと軽減策
""")
```

### Step 2: 画面設計

ユーザー向けの画面がある場合に実施する（CLI ツール等では省略可）。

#### 設計内容

- **画面一覧** — 全画面のリストと概要（ID 採番: `SC-001`, `SC-002`, ...）
- **画面遷移** — 画面間の遷移フロー
- **主要画面のワイヤーフレーム概要** — テキストベースで要素と配置を記述

ネイティブアプリの場合はさらに：
- ナビゲーション構造（タブ、ドロワー、スタック）
- ジェスチャー操作の定義
- プラットフォーム固有の UI パターン（iOS: HIG / Android: Material Design）

#### 対話の進め方

1. `docs/requirements/feature-list.md` の機能一覧から必要な画面を洗い出す
2. 画面一覧をユーザーに提示して過不足を確認
3. 画面遷移のフローを整理
4. 主要画面の構成要素を対話で詰める

### Step 3: API 設計

API を持つシステムの場合に実施する。

#### 設計内容

- **エンドポイント一覧** — ID、URL、メソッド、概要（ID 採番: `API-001`, `API-002`, ...）
- **認証方式** — API の認証・認可方式
- **共通仕様** — ページネーション、エラーレスポンス方針、バージョニング
- **リクエスト/レスポンス概要** — 主要なパラメータとレスポンス形式（概要レベル）

このフェーズでは一覧と共通仕様を確定する。各エンドポイントの型定義等の詳細は Phase 3 で個別ファイルに展開する。

#### サブエージェント活用

```
Task(subagent_type="api-designer", prompt="""
以下の要件に基づいて API 設計をレビューしてください：

要件: {要件サマリ}
API 設計案: {設計案}

以下を確認してください：
1. RESTful 設計原則への準拠
2. エンドポイントの過不足
3. 命名規則の一貫性
""")
```

### Step 4: データモデル設計

データを永続化するシステムの場合に実施する。

#### 設計内容

- **エンティティ一覧** — テーブル/コレクションとその概要
- **リレーション** — エンティティ間の関係（ER 図をテキストで表現）
- **テーブル定義** — 各エンティティの主要カラム、型、制約
- **インデックス設計** — 主要なインデックスとその目的

基本設計の段階でテーブル定義（カラム名、型、制約）まで含める。Phase 3 では必要に応じて個別テーブルの詳細ファイルに展開する。

#### サブエージェント活用

```
Task(subagent_type="data-modeler", prompt="""
以下の要件に基づいてデータモデルを評価してください：

要件: {要件サマリ}
データモデル案: {設計案}

以下を確認してください：
1. 正規化の適切さ
2. リレーションの妥当性
3. パフォーマンス上の懸念
""")
```

## ステップ選択の判断基準

| プロジェクトの種類 | 推奨ステップ |
|------------------|------------|
| Web アプリ（フルスタック） | Step 1-4 全て |
| ネイティブアプリ | Step 1-4 全て |
| API サーバーのみ | Step 1, 3, 4 |
| CLI ツール | Step 1 のみ（必要に応じて Step 4） |
| フロントエンドのみ | Step 1, 2 |
| データパイプライン | Step 1, 4 |

ユーザーと対話しながら、どのステップが必要か判断する。
