# 要件定義フェーズ（Phase 1）詳細手順

このドキュメントは `/design` スキルの Phase 1 で参照される。

## 目的

ユーザーとの対話を通じて、プロジェクトの全体像・用語・機能一覧・要件を明確化し文書化する。

## ステップ

### Step 1: プロジェクト概要のヒアリング

最初に大枠を把握する。AskUserQuestion で以下を聞く（一度に全部ではなく段階的に）：

- **何を作るのか** — プロダクト/機能の概要
- **誰のためか** — ターゲットユーザー
- **なぜ作るのか** — 背景・動機・解決したい課題
- **既存システムとの関係** — 新規構築か、既存への追加か
- **プラットフォーム** — Web / ネイティブアプリ（iOS/Android）/ デスクトップ / CLI 等

ユーザーの回答に応じて深掘り質問を追加する。
回答が揃ったら `docs/project-overview.md` を作成する。

### Step 2: 用語の定義

プロジェクト固有の用語を整理する。ドメイン知識のギャップを防ぎ、設計全体の一貫性を担保するために早期に行う。

- 対話の中で出てきた業務用語・ドメイン用語をピックアップする
- 「この用語はどういう意味で使っていますか？」と確認する
- 曖昧な用語があれば定義を明確化する

回答が揃ったら `docs/glossary.md` を作成する。設計の進行中も随時更新する。

### Step 3: 機能一覧の作成

システムが持つ機能を体系的にリストアップする。機能要件より先に「何ができるか」の全体像を把握する。

1. **機能カテゴリの洗い出し** — 「このシステムにはどんな機能グループがありますか？」
2. **各カテゴリの機能列挙** — カテゴリごとに具体的な機能を対話で洗い出す
3. **一覧の提示とフィードバック** — 中間リストを見せて過不足を確認

`docs/requirements/feature-list.md` に出力する。

### Step 4: 機能要件の洗い出し

機能一覧をもとに、各機能の詳細をユーザーと対話で明確化する：

1. **ユースケースの確認** — 各機能の具体的な利用シーンを聞く
2. **受け入れ条件の定義** — 「この機能が完成した状態とは？」
3. **優先順位付け** — Must / Should / Could（MoSCoW）で分類

対話のコツ：
- ユーザーが挙げた機能について「具体的にはどういう操作を想定していますか？」と掘り下げる
- ある程度出揃ったら中間リストを提示してフィードバックをもらう
- 全ての機能を同じ深さで掘り下げる必要はない — Must を優先的に詳細化する

### Step 5: 非機能要件の確認

ユーザーが意識していない非機能要件を引き出す。全項目を聞く必要はなく、プロジェクトに関連するものだけ確認する：

| カテゴリ | 確認ポイント |
|---------|------------|
| パフォーマンス | レスポンスタイム、同時接続数、処理量 |
| セキュリティ | 認証方式、データ暗号化、アクセス制御 |
| 可用性 | 稼働時間目標、障害復旧 |
| スケーラビリティ | 将来の拡張見込み |
| 保守性 | 技術制約、チーム体制 |
| 互換性 | 対応ブラウザ、OS、デバイス |
| 規制・コンプライアンス | 個人情報保護、業界規制 |

ネイティブアプリの場合はさらに：

| カテゴリ | 確認ポイント |
|---------|------------|
| オフライン対応 | オフライン時の動作要件 |
| プッシュ通知 | 通知の種類・頻度 |
| デバイス機能 | カメラ、GPS、生体認証等の利用 |
| ストア要件 | App Store / Google Play のガイドライン準拠 |

質問の仕方の例：
- 「パフォーマンスについて、特に気にしているポイントはありますか？例えば、ページの表示速度や同時アクセス数など」
- 「セキュリティ面で必須の要件はありますか？例えば、ログイン認証の方式や、扱うデータの機密レベルなど」

### Step 6: スコープの確定

対話の中で出てきた情報をもとに、スコープを明確化する：

- **In Scope** — 今回のプロジェクトに含めるもの
- **Out of Scope** — 今回は対象外とするもの
- **将来検討** — 今回は含めないが将来的に追加する可能性があるもの

### Step 7: ドキュメント作成

合意した内容をドキュメントに整理する。

## 出力フォーマット

### project-overview.md（プロジェクト概要）

```markdown
# プロジェクト概要

## 基本情報

- **プロジェクト名**: {名前}
- **作成日**: {YYYY-MM-DD}
- **最終更新日**: {YYYY-MM-DD}

## 概要

{プロジェクトの概要を 2-3 文で説明}

## 背景・目的

{なぜこのプロジェクトが必要か、何を解決するか}

## ターゲットユーザー

{誰のためのプロダクトか}

## プラットフォーム

{Web / iOS / Android / デスクトップ / CLI 等}

## スコープ

### In Scope
- {含めるもの}

### Out of Scope
- {含めないもの}

### 将来検討
- {将来的に追加する可能性があるもの}
```

### glossary.md（用語集）

```markdown
# 用語集

| 用語 | 定義 | 備考 |
|------|------|------|
| {用語} | {定義} | {補足} |
```

### feature-list.md（機能一覧）

```markdown
# 機能一覧

## {カテゴリ名}

| ID | 機能名 | 概要 | 優先度 |
|----|--------|------|--------|
| FT-001 | {機能名} | {概要} | Must |
| FT-002 | {機能名} | {概要} | Should |

## {カテゴリ名}

| ID | 機能名 | 概要 | 優先度 |
|----|--------|------|--------|
| FT-003 | {機能名} | {概要} | Must |
```

### functional.md（機能要件詳細）

```markdown
# 機能要件詳細

## FR-001: {機能名}

- **対応機能**: FT-001
- **優先度**: Must
- **概要**: {説明}
- **ユースケース**:
  1. {ユーザーが〜する}
  2. {システムが〜する}
- **受け入れ条件**:
  - [ ] {条件1}
  - [ ] {条件2}
- **備考**: {補足}

## FR-002: {機能名}
...
```

### non-functional.md（非機能要件詳細）

```markdown
# 非機能要件詳細

## NFR-001: {カテゴリ} — {要件名}

- **優先度**: Must
- **要件**: {具体的な要件}
- **計測基準**: {どう測定するか}
- **備考**: {補足}
```

## 対話の流れの例

```
Claude: まず、今回作るものの概要を教えてください。
       どんなシステム/機能を想定していますか？

User:  ECサイトのレビュー機能を追加したい

Claude: レビュー機能ですね。いくつか確認させてください：
       1. レビューは商品に対してつけるものですか？
       2. テキストだけでなく、画像や星評価も含みますか？

User:  商品に対して。星評価とテキストの両方

Claude: 承知しました。もう少し詳しく聞かせてください：
       - レビューの編集・削除はユーザー自身ができますか？
       - レビューに対する返信やいいね機能は必要ですか？

...（対話を続ける）
```

## 注意事項

- 一度に全てを聞こうとしない。自然な対話の流れで情報を引き出す
- ユーザーが「わからない」と言った項目は「将来検討」に分類し、ブロッカーにしない
- 技術的な判断（「REST vs GraphQL」等）はこのフェーズでは深入りしない — 基本設計で扱う
- 用語集は完璧を目指さない — 設計の進行に合わせて随時追加する
