# テスト設計

このドキュメントは `/design` スキルの拡張設計トラックとして、ユーザーの要望または AI の提案時に参照される。

## 目的

実装前にテスト戦略とテスト計画を策定し、品質基準を明確にする。
V字モデルの考え方に基づき、各設計フェーズの成果物に対応するテストレベルを定義する。

## 前提

- 要件定義（`feature-list.md`, `functional.md`）が存在すること
- 基本設計または詳細設計が進行中、または完了していること

## いつ実施するか

以下のいずれかに該当する場合に実施を検討する：

- ユーザーが「テスト計画を作りたい」「テスト方針を決めたい」と明示した場合
- 機能数が多く（目安: Must 機能 10 以上）、テスト戦略なしでは実装後の品質担保が困難な場合
- チーム開発で複数人がテストを書く場合
- 品質要件（非機能要件）が厳しいプロジェクト

## V字モデルとの対応

```
要件定義（Phase 1）  ←→  システムテスト（受入テスト）
基本設計（Phase 2）  ←→  結合テスト
詳細設計（Phase 3）  ←→  単体テスト
```

各設計フェーズの成果物がテスト設計のインプットになる。

## ステップ

### Step 1: テスト戦略の策定

プロジェクト全体のテスト方針を決める。

#### ヒアリング項目

- テストの自動化方針（どこまで自動化するか）
- 使用するテストフレームワーク・ツールの希望はあるか
- CI/CD でのテスト実行を想定しているか
- カバレッジ目標はあるか
- E2E テストは必要か

#### 設計内容

| 項目 | 定義する内容 |
|------|------------|
| テストレベル | 単体・結合・システム・受入の各レベルの定義と範囲 |
| テストツール | 各レベルで使用するフレームワーク・ライブラリ |
| 自動化方針 | 自動テストと手動テストの境界 |
| カバレッジ目標 | ライン/ブランチカバレッジの目標値 |
| テスト環境 | テスト用環境の構成（ローカル、CI、ステージング） |

#### 出力

`docs/testing/test-strategy.md` に記載する。

```markdown
# テスト戦略

## テストレベル

| レベル | 対象 | ツール | 自動化 | 実行タイミング |
|--------|------|--------|--------|-------------|
| 単体テスト | 関数・メソッド単位 | {ツール} | 自動 | コミット時（CI） |
| 結合テスト | モジュール間連携 | {ツール} | 自動 | PR 時（CI） |
| E2E テスト | ユーザーフロー | {ツール} | 自動 | デプロイ前（CI） |
| 受入テスト | 要件充足確認 | 手動 | 手動 | リリース前 |

## カバレッジ目標

| 対象 | 目標 | 備考 |
|------|------|------|
| ライン | {%} | {補足} |
| ブランチ | {%} | {補足} |

## 自動化方針

- {方針1: 例「ビジネスロジックは単体テスト必須」}
- {方針2: 例「主要ユーザーフローは E2E テストでカバー」}
- {方針3: 例「UI のスタイリングは手動確認」}

## テスト環境

| 環境 | 用途 | DB | 外部サービス |
|------|------|-----|------------|
| ローカル | 開発時の単体・結合テスト | {SQLite/Docker等} | モック |
| CI | PR マージ前の自動テスト | {構成} | モック |
| ステージング | E2E・受入テスト | {構成} | {実サービス/サンドボックス} |

## テストデータ方針

- {方針: 例「テストデータは Factory パターンで生成」}
- {方針: 例「本番データのサニタイズコピーは使用しない」}
```

### Step 2: テスト計画の作成

要件・設計に基づいて、何をどうテストするかを計画する。

#### 対話の進め方

1. `docs/requirements/feature-list.md` の機能一覧を読み込む
2. 各機能の優先度（Must/Should/Could）に応じてテスト密度を決める
3. 「この機能で特にテストが重要なポイントはどこですか？」と確認
4. テスト観点を整理してユーザーに提示

#### 設計内容

- 機能ごとのテスト観点（正常系・異常系・境界値）
- テスト優先度（Must 機能は厚く、Could 機能は薄く）
- テストケースの粒度目安

#### 出力

`docs/testing/test-plan.md` に記載する。

```markdown
# テスト計画

## テスト対象と優先度

| 機能 ID | 機能名 | テスト優先度 | テストレベル | テスト観点 |
|---------|--------|------------|------------|----------|
| FT-001 | {機能名} | 高 | 単体 + 結合 | {観点} |
| FT-002 | {機能名} | 高 | 単体 + 結合 + E2E | {観点} |
| FT-003 | {機能名} | 中 | 単体 | {観点} |

## 機能別テスト観点

### FT-001: {機能名}

| テストケース | 種別 | 期待結果 | レベル |
|------------|------|---------|--------|
| {正常系: 有効な入力で処理成功} | 正常系 | {期待結果} | 単体 |
| {異常系: 無効な入力でエラー} | 異常系 | {期待結果} | 単体 |
| {境界値: 最大文字数の入力} | 境界値 | {期待結果} | 単体 |
| {結合: API経由でデータ永続化} | 結合 | {期待結果} | 結合 |

### FT-002: {機能名}

...

## E2E テストシナリオ

| ID | シナリオ | ステップ | 期待結果 |
|----|---------|---------|---------|
| E2E-001 | {ユーザーフロー名} | {ステップ概要} | {期待結果} |
```

### Step 3: API テスト設計（API がある場合）

`docs/api/api-list.md` が存在する場合、API レベルのテスト観点を整理する。

#### 設計内容

- 各エンドポイントのテスト観点
- 認証・認可のテストパターン
- エラーハンドリングのテストパターン

#### 出力

`docs/testing/api-test-plan.md` に記載する。

```markdown
# API テスト計画

## エンドポイント別テスト観点

### API-001: {エンドポイント名}

| テストケース | メソッド | 期待ステータス | 観点 |
|------------|---------|-------------|------|
| 正常リクエスト | POST | 201 | 正常系 |
| 必須フィールド欠落 | POST | 400 | バリデーション |
| 未認証リクエスト | POST | 401 | 認証 |
| 権限不足 | POST | 403 | 認可 |
| 存在しないリソース | GET | 404 | 異常系 |
| 重複作成 | POST | 409 | 競合 |

## 共通テストパターン

### 認証テスト

| パターン | 期待結果 |
|---------|---------|
| トークンなし | 401 |
| 期限切れトークン | 401 |
| 無効なトークン | 401 |
| 他ユーザーのリソースへのアクセス | 403 |

### ページネーションテスト

| パターン | 期待結果 |
|---------|---------|
| デフォルトパラメータ | 1ページ目が返る |
| page=2 | 2ページ目が返る |
| 範囲外のページ | 空配列が返る |
| 不正な値 | 400 |
```

## 出力ディレクトリ

```
docs/testing/
├── test-strategy.md      ← テスト戦略
├── test-plan.md           ← テスト計画（機能別テスト観点）
└── api-test-plan.md       ← API テスト計画（API がある場合）
```

## `/preflight` との連携

テスト設計の成果物は `/preflight` でのタスク分解に直接活用できる：

- `test-strategy.md` → テスト環境構築タスクの根拠
- `test-plan.md` → 各機能のテストタスクの粒度と優先度
- `api-test-plan.md` → API テストの実装タスク

## 注意事項

- テスト設計は完璧を目指さない — Must 機能を中心にテスト観点を整理する
- テストケースの詳細（具体的なテストデータ、モックの実装方法等）は実装時に `tester` エージェントが担当する
- テスト戦略はプロジェクト初期に決め、テスト計画は設計の進行に合わせて段階的に作成する
- 過度に細かいテストケースは保守コストが高い — 重要なビジネスルールと境界値に集中する
